# Extraction Patterns

How to extract text from different source formats before converting to Obsidian markdown.

## Text-extractable PDFs (the common case)

Most modern PDFs — anything generated by Word, LaTeX, Google Docs, or modern Chinese textbook publishers — has a real text layer underneath. Use `pdftotext` from the `poppler-utils` package.

```bash
mkdir -p /tmp/pdf_extracts
pdftotext -layout "<source.pdf>" /tmp/pdf_extracts/<name>.txt
wc -l /tmp/pdf_extracts/<name>.txt
```

**Why `-layout`?** Preserves columns and tables visually. Without it, multi-column documents get scrambled.

**Quick health check** on the extracted text:
- Should be at least ~50 lines for any real document
- Open the first 100 lines and read — if it's recognizable prose, you're good
- If it's < 2 KB and mostly garbage, the PDF is image-only (see next section)

**Always prefer reading the `.txt`** over reading the PDF directly:
- Bypasses the 32 MB API attachment limit for large PDFs
- Much faster
- Easier to navigate with offset/limit if huge

**Common pdftotext artifacts to strip during conversion:**
- Standalone page numbers on their own line (`1`, `2`, `3`…) — leftover from page footers
- Repeated running headers ("吉屋教研组", "WWW.ZNOTES.ORG", etc.) — appear once per page
- Ligature breakdowns where the PDF font couldn't be decoded — e.g., `ﬁ`, `ﬂ`, or words split as `dif​ficulty` (with a zero-width joiner). Rejoin them.
- Bullet characters rendered as `•`, `o`, `■` — convert to `-` markdown bullets

## Image-only PDFs (scans, photos, slide decks)

If `pdftotext -layout` returns < 2 KB of mostly nothing, the PDF is image-based. Switch to the Read tool with the `pages` parameter (max 20 pages per call):

```
Read("<source.pdf>", pages="1-20")
```

For longer documents, read in chunks:
```
Read("<source.pdf>", pages="1-20")
Read("<source.pdf>", pages="21-40")
Read("<source.pdf>", pages="41-60")
```

**When the user already has image attachments:** if the source comes with companion PNG/JPG images (e.g., `attachments/page-1.png`, `attachments/page-2.png`), you can:
1. Read those images directly with `Read("<path>.png")` — they render visually in your context.
2. In the converted markdown, embed them with `![[attachments/page-N.png]]` so they appear in the rendered Obsidian note alongside your transcription.

**Slide decks (LaTeX Beamer, PowerPoint export to PDF)**: usually text-extractable. Treat as text PDFs first.

## DOCX

```bash
pandoc -t markdown -o /tmp/pdf_extracts/<name>.md "<source.docx>"
```

If pandoc isn't installed, fall back to LibreOffice headless:
```bash
soffice --headless --convert-to txt --outdir /tmp/pdf_extracts "<source.docx>"
```

If neither is available, tell the user and ask how they'd like to proceed (often they'll just copy-paste the text).

## PPTX

The `anthropic-skills:pptx` skill exists specifically for `.pptx` files. Use it for full PowerPoint reading. For a quick text-only extract:
```bash
pandoc -t markdown -o /tmp/pdf_extracts/<name>.md "<source.pptx>"
```

## Chinese / CJK PDFs

The user's vault has many Chinese-language PDFs. A few CJK-specific gotchas:

**Punctuation:** `pdftotext` usually preserves Chinese punctuation correctly (，。；："" etc.). Keep them as-is — don't replace with ASCII equivalents.

**Half-width vs full-width:** Numbers and Latin letters inside Chinese text are often full-width (`１`, `Ａ`). Keep them as the source has them unless they're clearly OCR noise.

**Fragmented glyphs:** older PDFs sometimes split Chinese characters in weird ways. If you see broken-looking single characters or radical fragments, check the original PDF page with Read to confirm before transcribing.

**Vertical text:** some classical Chinese poetry is typeset vertically. `pdftotext` often scrambles it. For these, prefer Read on the PDF page directly.

## Source files in the user's vault

The user has many archived PDFs. Common locations to check:
- `<subject>_<code>/<topic>/attachments/` — companion PDFs and images for individual notes
- `<subject>_<code>/99XXXothers/` — bulk archives, often big PDFs unrelated to current study
- `50_permanent/attachments/` — MD5-hashed embedded files (referenced via `![[hash_MD5.pdf]]`)

When asked to "convert this PDF", confirm the path before starting — there are often multiple files with similar names.

## Large file handling

If a PDF is > 100 pages or > 20 MB:
1. Always extract with `pdftotext -layout` first — never try to read the PDF directly.
2. After extraction, use `Read` with `offset` and `limit` to navigate the `.txt` in chunks rather than loading the whole thing at once.
3. For very large source docs, consider splitting the output across multiple notes (one per chapter / topic) rather than producing a single 200 KB markdown file.

## Verifying the extraction

Before you start writing the markdown note, sanity-check the extraction:
- `wc -l` the file. Does the line count match roughly what you'd expect from the page count? (Rule of thumb: ~30-50 lines per page for prose, 10-30 for slides.)
- `head -50` the file. Is the start of the document recognizable?
- `tail -30` the file. Is the end of the document there, or did extraction cut off?
- Grep for known section headers. Are they all present?

If extraction is incomplete or garbled, fix it before writing the note. Don't paper over bad extraction with paraphrasing.
